## Issue Detail

### Issue 1 — `configuration-patterns.md`: database name uses `estate_archive`

**Where**: Config examples throughout the file (TypeScript section lines ~260, 278, 945, 963;
Python section lines ~580, 1086).

**Problem**: The project was rebranded to "Institutional Knowledge" on 2026-02-26. All active
documentation was updated at that time, but the config example files inside this skill still
use `estate_archive` and `estate_archive_prod` as database names. This is inconsistent with
the rebrand and gives a misleading signal to implementers about naming conventions.

**Fix**: Change `estate_archive` → `institutional_knowledge` and `estate_archive_prod` →
`institutional_knowledge_prod` in all config file examples within the skill.

**Files to change**: `.claude/skills/configuration-patterns.md`

**user review**: accepted change

### Issue 2 — `configuration-patterns.md`: Docling presented as HTTP API

**Where**: Python section — `DoclingAdapter` class (approx lines 475–515).

**Problem**: The `DoclingAdapter` example makes HTTP requests to a Docling API at
`http://localhost:8000`. ADR-011 specifies Docling as the primary OCR engine used as a
**Python library**, not as a network service. The Tesseract adapter is the fallback.

The skill inverts the relationship: it presents Docling as the API-based approach and
Tesseract as the simpler local approach. An implementer reading this literally may try to
set up a Docling HTTP service that does not exist in the architecture.

**Fix**: Add a clarifying note at the top of the DoclingAdapter example stating: "This
example illustrates the HTTP adapter pattern. In the real implementation, Docling is used
as a Python library (installed via pip), not as a network service. The HTTP pattern shown
here applies to remote API providers such as commercial OCR services." Alternatively,
rewrite the DoclingAdapter to use the Docling library directly with a comment explaining
the library-vs-API distinction.

**ADR reference**: ADR-011

**Files to change**: `.claude/skills/configuration-patterns.md`

**user review**: accepted change

### Issue 3 — `dependency-composition-pattern.md`: C3 Python context implicit

**Where**: "Why This Matters" section (approx lines 35–46).

**Problem**: The skill states Institutional Knowledge has three deployment targets: Express
HTTP service, FastAPI HTTP service (processing pipeline admin), and MCP server. C3 (Query

**Fix**: Add one sentence to the "Why This Matters" section noting that C3 is also
implemented in Python (alongside C2), and that the Python handler function pattern in this
skill applies directly to C3 route handlers and any future C3 MCP integration.

**Files to change**: `.claude/skills/dependency-composition-pattern.md`

**user review**: accepted change

### Issue 4 — `pipeline-testing-strategy.md`: `dimensions: 384` hardcoded in fixtures

**Where**: `expected-outputs.json` fixture example (approx lines 168–174).

**Problem**: The expected outputs fixture specifies `"dimensions": 384` for the embedding
step. ADR-024 establishes that embedding dimensions are config-driven and not hardcoded —
the dimension is determined by the configured embedding model. A value of 384 corresponds
to a specific model (all-MiniLM-L6-v2); the actual Phase 1 model has not been chosen yet.
If a different model is selected at implementation, the fixture value would be wrong from
day one.

**Fix**: Replace `"dimensions": 384` with a comment-annotated placeholder:

```json
"embedding": {
  "dimensions_min": 64,
  "dimensions_note": "Actual dimension depends on configured embedding model — update when model is selected",
  "value_type": "float32"
}
```

Or add a note in the fixture example directing the implementer to check the configured model's
output dimension rather than relying on a hardcoded value.

**ADR reference**: ADR-024

**Files to change**: `.claude/skills/pipeline-testing-strategy.md`

**user review**: add a note and make it clear the example is just an example

### Issue 5 — `pipeline-testing-strategy.md`: integration test asserts exact dimension count

**Where**: Integration test example `test_embedding_produces_correct_shape` (approx lines
229–233).

**Problem**: The integration test asserts:

```python
assert len(result["embedding"]) == expected["dimensions"]
```

This is tied to the hardcoded `384` value in Issue 4 above. If the dimension is wrong in
the fixture, this assertion fails. More fundamentally, the assertion couples the test to a
specific model choice, contradicting the provider-agnostic configuration principle.

**Fix**: Change the assertion to:

```python
assert len(result["embedding"]) > 0  # Dimension determined by configured provider
```

This mirrors the fix already applied to `embedding-chunking-strategy.md` in the previous
session.

**ADR reference**: ADR-024

**Files to change**: `.claude/skills/pipeline-testing-strategy.md`

**user review**: accepted change

### Issue 8 (Critical) — `metadata-schema.md`: graph rebuild includes `llm_extracted`

**Where**: "Graph Rebuild Trigger" section under "Curation Workflow" (approx lines 419–426).

**Problem**: The skill states the graph rebuild reads:

```python
source IN ('candidate_accepted', 'llm_extracted')
```

This is wrong. ADR-039 and the architecture document (`documentation/project/architecture.md`,
"Graph Rebuild" section) both state the rebuild reads:

```sql
vocabulary_terms WHERE source IN ('seed', 'manual', 'candidate_accepted')
```

`llm_extracted` entities that are still in the review queue are explicitly **excluded** from
the graph until the curator accepts them. This is a stated design intention: the graph only
contains evidentially accepted entities, not unreviewed LLM outputs. Including `llm_extracted`
would bypass the human-in-the-loop curation gate.

**Fix**: Change the source list in the graph rebuild section from:

```python
source IN ('candidate_accepted', 'llm_extracted')
```

to:

```python
source IN ('seed', 'manual', 'candidate_accepted')
```

Also update the plain-English description above the code to match.

**ADR references**: ADR-039, ADR-014

**Files to change**: `.claude/skills/metadata-schema.md`

**user review**: accepted change
