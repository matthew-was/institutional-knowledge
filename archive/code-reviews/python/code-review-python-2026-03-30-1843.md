## Findings

### Blocking

**B-1 — Missing skeleton files required by the plan module structure**

`services/processing/`

The plan module structure (`senior-developer-python-plan.md`, lines 40–72) specifies
several Python source files that must be created as stubs in Task 1. The acceptance
condition says "the directory tree matches the structure in the plan." The following files
are absent:

- `pipeline/orchestrator.py` — shown explicitly in the plan tree
- `query/router_factory.py` — shown explicitly in the plan tree
- `query/query_understanding.py` — shown explicitly in the plan tree
- `query/context_assembly.py` — shown explicitly in the plan tree
- `query/response_synthesis.py` — shown explicitly in the plan tree
- `query/query_handler.py` — shown explicitly in the plan tree
- `shared/http_client.py` — shown explicitly in the plan tree
- `shared/config.py` — shown explicitly in the plan tree
- `tests/test_app.py` — the plan notes "tests/ pipeline/, query/, shared/ subdirectories;
  test_app.py"

These are skeleton/stub files — no logic is required. Each needs only a module docstring
(with ADR citation where applicable) and can otherwise be empty. Their absence means the
directory tree does not match the plan and the acceptance condition is not met.

**B-2 — `tests/` directory is missing `__init__.py`**

`services/processing/tests/`

`development-principles-python.md` (Package Structure section) states: "`tests/` and all
its subdirectories (avoids pytest import ambiguity)." The four subdirectories under
`tests/` (`pipeline/`, `query/`, `shared/`, `fakes/`) each have an `__init__.py`. The
`tests/` directory itself does not. Without it, pytest may encounter import ambiguity
errors when the test suite grows, and the acceptance condition ("zero errors") cannot be
verified to hold as tests are added. Add an empty `tests/__init__.py`.

**B-3 — `docling` dependency is commented out in `requirements.txt`**

`services/processing/requirements.txt`, line 9

The task description specifies that `requirements.txt` must list `docling`. The file
contains `# docling` (commented out). Task 1's description is explicit: the file must
list `docling`. Even if the package is difficult to install in the current environment,
it must remain uncommented in `requirements.txt` because this file defines the service's
runtime dependencies. If docling is excluded from installation for local testing, that
belongs in a local override or a note in developer docs — not in a commented-out line in
the canonical requirements file.

### Suggestions

**S-1 — `pyproject.toml` targets Python 3.13 while the plan specifies Python 3.11+**

`services/processing/pyproject.toml`, line 2: `target-version = "py313"`

The plan (`development-principles-python.md`, Technology Constraints) specifies
"Python 3.11+ with type annotations." The `target-version` in `pyproject.toml` drives
ruff's syntax understanding. Setting it to `py313` means ruff will not warn if 3.13-only
syntax is used, which could cause issues if the deployment environment runs 3.11 or 3.12.
Consider aligning `target-version = "py311"` with the stated minimum version requirement.
(Note: this is a suggestion, not blocking, because the `.python-version` file and
`Dockerfile` control the actual runtime version, and both the current dev environment and
the Docker image appear to use Python 3.13.)

**S-2 — `ci_integration` marker registered but `asyncio_mode = auto` added without
a corresponding `pytest-asyncio` entry in `requirements-dev.txt`**

`services/processing/pytest.ini`, line 2: `asyncio_mode = auto`

`pytest-asyncio` controls `asyncio_mode`. It is listed in `requirements-dev.txt` but only
as a bare package name without a version pin. This is fine for now as a scaffold. The
`ci_integration` marker is a forward-looking addition that goes beyond what Task 1
specifies. This is acceptable (and matches the testing strategy), but both the marker
and `asyncio_mode` belong to the same concern — consider adding a comment in `pytest.ini`
linking the `ci_integration` marker to `development-principles-python.md` for clarity.

**S-3 — `shared/generated/__init__.py` has an ADR citation but other `__init__.py`
files do not**

`development-principles-python.md` says "`__init__.py` files are empty markers unless
the package warrants a module docstring (e.g. `shared/generated/__init__.py` cites
ADR-048)." The `shared/generated/__init__.py` correctly cites ADR-048. No other
`__init__.py` needs a docstring at this stage — this observation is noted for completeness
and no action is required.
