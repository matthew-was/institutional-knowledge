## Acceptance condition

**Condition type**: automated

Integration tests against a real PostgreSQL instance confirm:

- (a) `writeEntity` + `getEntity` round-trip: insert an entity with a known ID; retrieve it; verify fields match. An entity with no `entity_document_occurrences` is not returned by `getEntity`.
- (b) `writeRelationship` + `getRelationships`: insert two entities and a relationship; retrieve outgoing relationships from the source entity; verify result.
- (c) `traverse` depth 1, 2, 3: build a three-hop chain; verify that traversal with maxDepth=1 returns only depth-1 neighbours, maxDepth=2 returns depth-1 and depth-2, and so on.
- (d) `findEntitiesByType` filtering: insert entities of two categories; verify only the correct category is returned.
- (e) `findDocumentsByEntity` join: insert entity, document, and occurrence; verify the returned document reference has the correct description and date.
All integration tests pass.

**Result**: Met

- (a) Two tests cover this condition. The first confirms the full round-trip: entity written, getEntity returns null before any occurrence, then returns the entity with correct fields after an occurrence is inserted. The second confirms that an entity written with no occurrence always returns null. A third test covers the upsert-on-conflict path.
- (b) Three tests cover this condition: outgoing relationships, incoming relationships, and duplicate-insert idempotency. The duplicate test also confirms that exactly one row is stored after two identical writes.

- (d) Two tests: one confirms that only the correct category is returned when multiple categories exist; the second confirms that entities without occurrences are excluded from `findEntitiesByType` results.

### Suggestions

**S-001 — `apps/backend/src/db/tables.ts`, line 13: stale "randomUUID" in doc comment**

The comment at line 13 reads:

```text
*   - id           — generated by application code (randomUUID) before insert,
```

Line 19 of the same comment correctly says "uuidv7()". The task description notes this file was changed to update the comment from `randomUUID` to `uuidv7()`, but line 13 was not updated. This is a doc-only inconsistency with no runtime effect; worth fixing for accuracy.

**S-002 — `apps/backend/src/db/repositories/graph.ts`, lines 112–120: `void` used to suppress query builder mutation warnings**

The `findRelationships` method uses `void query.where(...)` to discard the return value of the Knex query builder's `.where()` call while relying on the in-place mutation of `query`. This works correctly — Knex's `QueryBuilder.where()` mutates the builder and returns the same object — but the `void` idiom is unusual here. The alternative is a fluent chain assigned back to `query` (e.g. `query = query.where(...)`, which requires `let` instead of `const`) or a conditional ternary. The current approach is not a bug, but it may confuse a reader who expects `void expr` to mean "side-effect only, result intentionally ignored" rather than "mutation already applied to the object in scope". Consider a brief inline comment explaining that `.where()` mutates `query` in place, making the `void` discard safe.

**S-003 — `apps/backend/src/graphstore/PostgresGraphStore.ts`, line 38: `normalisedTerm` computed without punctuation stripping**

`writeEntity` computes `normalisedTerm` as `entity.term.toLowerCase()`. ADR-028 describes the normalised term as "lowercase, punctuation stripped". The migration (`20260303000002_create_vocabulary.ts`) defines `normalised_term` as a plain `text` column (not a database-generated column), so the normalisation is entirely application-side. The inconsistency between ADR-028's description and the actual normalisation applied here exists both in this file and in the earlier migration design; it predates Task 7. No change is required by Task 7, but it is worth noting so that the processing handler (Task 11+) that also computes `normalised_term` applies the same logic consistently. Consider filing a follow-up note or updating the `writeEntity` comment to document that punctuation stripping was not implemented, and that the processing handler must match this behaviour.

**S-004 — `apps/backend/src/graphstore/PostgresGraphStore.ts`, line 136: `confidence` hardcoded to `null` in `traverse()` result**

The `TraversalRawRow` type (and the CTE's `SELECT DISTINCT` projection) does not include `confidence`, so it cannot be populated from the raw query result. The `confidence: null` assignment is the only correct option given the current schema. A brief inline comment explaining why (e.g. "confidence is not projected by the CTE — not available in traversal results") would save a future reader from wondering whether this was an oversight.
